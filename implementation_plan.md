# 前端开发执行计划 - Horizon Travel Agent v1

本计划将根据 PRD 和用户体验分析，拆解 Horizon Travel Agent 平台第一阶段（v1）的各项前端增强任务。

## 第一阶段：规划页与反馈体系 (Planner & Feedback)

### 任务 1.1：规划页软约束引导
- **目标**：在不强行拦截的前提下，委婉告知用户 3-7 天以及单目的地的 AI 偏好。
- **详情**：
    - 在目的地输入框附近增加“AI 建议”提示（如：建议输入单个城市以获得最佳效果）。
    - 根据日期选择自动计算旅行天数，若不在 3-7 天范围内，显示非阻塞式的琥珀色提示语。
    - 视觉风格：使用细腻的、符合 Premium 调性的警示色，保持界面优雅。

### 任务 1.2：用户评价与反馈系统
- **目标**：收集用户对 AI 生成行程质量的真实评价。
- **详情**：
    - 开发可复用的 `FeedbackForm` 组件（星级评分 + 评论文本框）。
    - 在行程详情页（`ItineraryPage`）底部集成该组件。
    - 模拟 API 提交逻辑。

### 任务 1.3：角色管理与权限保护 (RBAC)
- **目标**：建立“用户”与“管理员”的角色隔离机制。
- **详情**：
    - 在 `App.tsx` 引入 `userRole` 状态。
    - 移除全局浮动的后台切换按钮，改为基于角色的 Navbar 入口显示。
    - 实现简单的视图守卫，防止非管理员进入 `dashboard`。

## 第二阶段：行程详情与生成体验 (Itinerary UX)

### 任务 2.1：每日预算与开销明细
- **目标**：响应 PRD 需求，提供更细颗粒度的财务数据展示。
- **详情**：
    - 在每日行程标题旁增加“日预算”标签。
    - 更新 `ActivityCard`，展示单个活动项的“预估人均开销”。
    - 确保侧边栏的总预算逻辑能正确汇总这些数据。

### 任务 2.2：“修改偏好并重生成”流程
- **目标**：允许用户在不重填整个表单的情况下微调行程。
- **详情**：
    - 在详情页侧边栏增加“微调偏好”按钮。
    - 弹出抽屉或模态框，包含兴趣点和节奏的选择器。
    - 触发带参数的重新生成状态。

### 任务 2.3：智能 Agent 思考状态加载页
- **目标**：让等待时间变得更具“智能感”和生产力感。
- **详情**：
    - 用一系列动态状态消息替换现有的简单 Loader。
    - 消息序列示例：“正在分析当地天气...” -> “正在根据您的口味筛选高分餐厅...” -> “正在优化步行路径...”。
    - 实现消息之间的平滑切换效果。

## 第三阶段：历史记录与后台打磨 (History & Admin)

### 任务 3.1：历史页快速操作
- **目标**：提升行程历史视图的实用性。
- **详情**：
    - 在 `TripCard` 上增加显眼的“重新生成”快捷按钮。
    - 区分“查看详情”和“重生成”的视觉权重。

### 3.2：管理后台增强
- **目标**：补全管理员查看反馈和错误日志的需求。
- **详情**：
    - 在仪表盘（`DashboardPage`）增加“最新反馈”卡片。
    - 实现失败任务的详情预览弹窗（显示原始 Prompt + 错误堆栈）。

### 3.3：导出预览模态框
- **目标**：优化导出功能的“占位”体验。
- **详情**：
    - 增加“生成预览中...”的中间状态，随后展示行程的文本大纲。
    - 在预览页提供最终的“下载”按钮。

## 第四阶段：移动端与最终润色

### 任务 4.1：移动端响应式审计
- **目标**：确保所有设备上的一致性高级体验。
- **详情**：
    - 修复详情页和仪表盘侧边栏在小屏幕上的堆叠显示。
    - 优化移动端点击目标的尺寸。

## 第五阶段：真实后端联调与鉴权系统 (Backend Integration & Auth)

### 任务 5.1：登录页与鉴权流 (Auth UX)
- **目标**：实现 PRD 5.1 节中规定的登录/注册页面及鉴权闭环。
- **详情**：
    - 开发 `LoginPage` (`/login`)，包含账号密码及第三方登录 UI 骨架。
    - 接入真实身份验证服务（如 Supabase Auth），彻底替换 `App.tsx` 中模拟的 `userRole` 状态。
    - 强化路由守卫（如 `RequireAuth` 组件），确保未登录用户强制跳转登录，并限制 Admin 专属页面访问。

### 任务 5.2：前端 API 服务层封装 (API Client)
- **目标**：根据 PRD 接口草案，搭建与后端交互的基础通道，准备替换现有的 mock 数据。
- **详情**：
    - 创建 `src/services/api` 目录，封装基础 HTTP 请求（如 axios 实例），统一处理 Token 注入与全局错误。
    - 对接 PRD 第 8 节定义的 API 规范（如 `POST /api/trips/plan`, `GET /api/history` 等）。
    - 针对大模型生成行程的长耗时任务，设计轮询 (Polling) 或 Server-Sent Events (SSE) 接收逻辑。

### 任务 5.3：全局状态与缓存管理升级 (State Management)
- **目标**：提升生产环境下的数据加载、缓存和同步体验。
- **详情**：
    - 引入 `@tanstack/react-query` 或类似库管理服务端状态。
    - 重构历史记录、行程详情等页面的数据抓取逻辑，实现平滑的数据缓存、重新验证和优雅的加载/错误回退 UI。

## 第六阶段：PRD 规划的可选增强 (Optional Enhancements)

### 任务 6.1：热门目的地与社交分享
- **目标**：落实 PRD 中的“可选增强”需求，提升平台运营属性。
- **详情**：
    - 在 Admin Dashboard 或前台入口增加“热门目的地排行”视图。
    - 探索基于 Canvas (如 `html2canvas`) 的“导出为分享长图”功能，补全现有的导出能力。

---

## 第七阶段：后端核心与 AI 编排 (Backend & AI Agent)

### 任务 7.1：Node.js 后端基础服务搭建
- **目标**：实现 PRD 第 6 节要求的 API 支撑。
- **详情**：
    - 在根目录搭建 Express 或 NestJS 服务环境。
    - 在 Supabase 中按照 `PRD.md` 落地 `trip_plans`、`itinerary_days` 等数据表。
    - 实现 API 基础路由与 Supabase 的 CRUD 交互。

# 第八阶段：鉴权增强与微信登录 (Auth Enhancement & WeChat Support)

### 任务 8.1：微信 OAuth 流程对接
- **目标**：支持中国区用户常用的微信扫码登录。
- **详情**：
    - 后端：实现微信 `code` 换取 `access_token` 及用户信息的逻辑。
    - 数据库：建立 `profiles` 表关联微信 ID。
    - 前端：添加微信登录按钮，处理重定向与 Magic Link 登录流。

### 任务 8.2：游客访问模式 (Guest Mode)
- **目标**：降低使用门槛，允许游客预览功能。
- **详情**：
    - 移除规划页（PlannerPage）的强制登录拦截。
    - 实现游客状态下的功能限制弹窗。
